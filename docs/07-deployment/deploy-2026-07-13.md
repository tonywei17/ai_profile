# 部署日志 — 2026-07-13

将 Hivision 从 Google Cloud Run（东京）切换到阿里云 ECS 自建容器，顺带把
`aiidphoto-backend` 的环境变量管理方式从"只存在于 PM2 内存里"重建为标准的
`ecosystem.config.js`。**切换过程中发生了一次真实生产中断**，根因是本 session
更早一次 `rsync --delete` 同步误删了服务器专属目录，本文档完整记录诊断与恢复
过程，并修正了导致该问题的标准部署命令。

- 阿里云 ECS：`8.148.158.232`（广州），与本项目共享该机器的还有其他 15+ 个 PM2

## 变更一：Hivision 切换到自建容器

```
HIVISION_URL: http://<Google Cloud Run 地址>  →  http://127.0.0.1:8899
```

只改这一个环境变量，Hivision 调用协议本身不变（`callHivision()` 代码无需改动）。

## 变更二：`ecosystem.config.js` 取代裸 PM2 env

之前 `aiidphoto-backend` 的环境变量只存在于 PM2 进程的内存态 `pm2_env` 里，没有
任何文件记录，改动全靠 `pm2 restart --update-env` 这种一次性操作——不可追溯、
不可复现。这次从 `pm2 jlist` 导出的真实运行值，重建成标准的
`/opt/aiidphoto-backend/ecosystem.config.js`（`exec_mode: "fork"`，权限 600，
**只存在服务器上，不进 git**——因为里面是真实密钥）。

完整的 27 个变量名（非值，来自 `backend/src/config.ts` 实际读取的
`process.env.*`，逐一核对无遗漏）：

REQUIRE_APP_KEY, REFERRAL_HMAC_SECRET, DAILY_BUDGET, WECHAT_APP_ID,
WECHAT_APP_SECRET, SESSION_TOKEN_SECRET, WECHAT_MCH_ID, WECHAT_SERIAL_NO,
WECHAT_API_V3_KEY, WECHAT_KEY_PATH, WECHAT_PLATFORM_PUBLIC_KEY_PATH,
WECHAT_PLATFORM_SERIAL_NO, WECHAT_NOTIFY_URL, WECHAT_VIRTUAL_PAY_ENABLED,
WECHAT_VIRTUAL_OFFER_ID, WECHAT_VIRTUAL_SANDBOX_APP_KEY,

（`TRIPO_API_KEY`、`HOST` 在旧的 `pm2_env` 里本来就不存在，代码里各自有默认值

切换前把旧 `pm2_env` 完整备份到服务器 `/root/aiidphoto-backend-env-backup-<时间戳>.json`
（root-only, 600），万一需要回滚可直接取用其中的原始 `HIVISION_URL` 等值。

## 事故记录：切换引发的生产中断

### 现象

`pm2 delete aiidphoto-backend && pm2 start ecosystem.config.js` 冷启动后，进程
反复重启（PM2 判定 "too many unstable restarts"，状态 `errored`）。服务器本机
`curl 127.0.0.1:9528/health` 连接失败，公网 `https://aiphoto-cn.foyli.cloud/health`
返回 502。

### 排查过程（记录两次错误假设，避免以后重复踩坑）

1. **怀疑 `exec_mode: cluster` 的 IPC 问题**——切到 `fork` 模式，重启，依然
   崩溃循环，排除。
2. **怀疑 PM2 守护进程状态损坏**——无直接证据，为了绕开 PM2 本身的干扰，改用
   直接 `node dist/index.js` 裸跑（用备份的 env 值、换端口避免冲突），5 秒内
   进程即退出，日志显示真实报错：
   ```
   Error: WECHAT_KEY_PATH does not exist
       at validateProductionConfig (config.js:88)
   ```
   这才是真正的根因，前两次假设都是错的。

### 根因

本 session **更早**为推送 sharp 修复而执行过一次：

```bash
rsync -av --delete --exclude node_modules --exclude .env \
  ./ root@8.148.158.232:/opt/aiidphoto-backend/
```

`certs/`（微信支付证书）和 `data/`（`payment-store.json` 订单记录）这两个目录
在 `backend/.gitignore` 里，本地从未存在过。`--delete` 参数会把"目标存在、源
不存在"的文件当作多余项删除——这两个服务器专属目录因此被误删。旧的 PM2 worker
当时仍在内存里跑着，没有重新校验证书文件，问题一直没暴露，直到这次为了切换
Hivision 而 `pm2 delete` 触发冷启动，才在证书校验这一步崩溃。

### 恢复

```bash
# 服务器上找到的历史备份(2026-06-12)
ls /opt/aiidphoto-backend-rollback-signfix-20260612-145000/certs/
#   apiclient_key.pem  apiclient_cert.pem  wechatpay_public_key.pem

mkdir -p /opt/aiidphoto-backend/certs
cp /opt/aiidphoto-backend-rollback-signfix-20260612-145000/certs/*.pem \
   /opt/aiidphoto-backend/certs/
chmod 600 /opt/aiidphoto-backend/certs/*.pem

pm2 delete aiidphoto-backend
pm2 start ecosystem.config.js
pm2 save
```

`data/` 无需恢复——产品从未正式运营，无真实订单数据；`PaymentStore.persist()`
自带 `fs.mkdirSync(..., {recursive:true})`，下次写入时自动重建。

### 影响范围

全程仅 `aiidphoto-backend` 一个 PM2 进程受影响。同机其余 15+ 个生产服务
（`dashboard-*`、`guixiaoshe-*`、`heninyishi-*`、`hotel-translation-*`、
`foyli-tech-website` 等）PID、uptime、重启次数在事故前后完全未变化。

### 已同步修复的文档 bug

`docs/07-deployment/cloud-run-deploy.md` 里标准部署命令的 `rsync --delete`
**缺少 `--exclude certs --exclude data`**，是这次事故能够发生的根本原因——已在
本次一并修正该文档，以后任何人按文档操作都不会再复现同样的问题。
