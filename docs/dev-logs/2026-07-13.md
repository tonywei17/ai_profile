## 完成任务

### 1. 修复 Hivision 透明通道 / CMYK 崩溃（sharp 归一化）

- 根因：Hivision matting 步骤的 `cv2.split()` 硬编码假设输入是 3 通道；手机截图等
  **RGBA PNG** 或 **CMYK JPEG** 会导致 `too many values to unpack` 崩溃，用户端表现
  为该类图片永远生成失败。
- 修复：`backend/src/routes/gemini.ts` 的 `callHivision()` 在发给 Hivision 前统一
  `sharp(buffer).flatten({background:"#ffffff"}).toColorspace("srgb").jpeg({quality:95})`，
  确保送进去的一定是纯 3 通道 sRGB JPEG。
- 新增依赖 `sharp@^0.33.5`。

### 2. Hivision 自建迁移（Google Cloud Run → 阿里云 ECS 自建容器）

- 之前：`HIVISION_URL` 指向 Google Cloud Run（东京），跨云调用，偶发冷启动拖慢生成。
- 现在：Hivision 以 Docker 容器（`linzeyi/hivision_idphotos`，`restart: unless-stopped`）
  自建在 `aiidphoto-backend` 所在的同一台阿里云 ECS 上，容器名 `hivision-api`，
  端口映射 `127.0.0.1:8899 -> 8080`，`HIVISION_URL` 改为 `http://127.0.0.1:8899`。
- 顺带把后端进程的环境变量管理方式，从"只存在于 PM2 内存里的 `pm2_env`、靠
  `--update-env` 修改"，重建为标准的 `ecosystem.config.js`（27 个变量，逐一核对
  `backend/src/config.ts` 实际读取的 key，完整无遗漏）。此文件**只存在服务器上**
  （含真实密钥，不适合入库），仓库侧的变量清单与模式记录见部署文档。

### 3. 切换引发的生产中断 ⚠️ 及修复

- 现象：`pm2 delete` 旧进程、按新 `ecosystem.config.js` 冷启动后，进程反复重启
  （"too many unstable restarts"），服务器本机与公网 `/health` 均不可达。
- 排查过程中依次排除了两个错误假设：`exec_mode: cluster` 的 IPC 问题（切到 `fork`
  后依然崩溃循环）、PM2 守护进程状态损坏。最终用直接 `node dist/index.js`（绕开
  PM2）裸跑，才看到真实报错：`Error: WECHAT_KEY_PATH does not exist`
  （`validateProductionConfig()`，`config.ts`）。
- 真正根因：本 session 更早为推送 sharp 修复而执行的 `rsync -av --delete
  --exclude node_modules --exclude .env` 同步，把服务器上 **从未入库、也不该入库**
  的 `certs/`（微信支付证书）和 `data/`（`payment-store.json` 订单记录）当作"本地
  没有的多余文件"删掉了——这两个目录在 `backend/.gitignore` 里，本地从未存在过。
  旧进程当时还在内存里跑着没有触发，直到这次冷启动重新校验证书文件才暴露。
- 修复：从服务器上 `/opt/aiidphoto-backend-rollback-signfix-20260612-145000/certs/`
  这份 6 月 12 日的历史备份恢复三个证书文件，干净重启。`data/` 无需恢复——产品
  从未正式运营，无真实订单数据；该目录已在下次写入时自动重建
  （`PaymentStore.persist()` 自带 `mkdirSync`）。
- 影响范围确认：全程仅 `aiidphoto-backend` 一个进程受影响，同机其余 15+ 个 PM2
  托管的生产服务（含 dashboard/guixiaoshe/heninyishi/hotel-translation 等）
  PID、uptime、重启次数全程未变化。

  `--exclude certs --exclude data`**，是这次中断的根本原因，已在本次一并修复文档
