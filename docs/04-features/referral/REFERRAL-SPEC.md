# 紹介キャンペーン(リファラル)機能規格 — 規約の事実根拠

> 本ドキュメントの目的:`CAMPAIGN-TERMS-ja.md` / `CAMPAIGN-TERMS-zh.md` に書く数字・条件が、実装事実と一致していることを保証するための「事実の一次ソース」。
> 規約の文言を変更する場合、または下記の数値・仕様を変更する場合は、**両方を同時に更新する**こと(景表法上の「表示と実装の一致」原則、jp-ad-compliance skill 参照)。
>
> 実装本体(バックエンド/iOS のコード)は本タスクの対象外(コード変更なし)。ここに書く数値は、本機能の設計 JSON(2026-07-22 時点の workflow 出力)に基づく想定仕様であり、実装完了後に本ドキュメントで実値と突合すること。

## 1. 特典の内容(規約 第4条の根拠)

| 対象者 | 条件 | 特典 | 上限・回数制限 |
|---|---|---|---|
| 招待された人(コード入力者) | 紹介コードを1回入力(成立) | Pro品質の生成 **3回分** | 1端末につき生涯 **1回のみ**成立(`MAX_REDEEMS_PER_DEVICE = 1`) |
| 招待した人(コード発行者) | 自分のコードで招待が1件成立するごと | 生成 **3回分**(成立ごとに加算) | 上限なし(ただし下記の不正防止措置あり) |
| 招待した人(コード発行者) | 累計 **3件**成立時点 | **3日間の Pro 体験**(1アカウント **生涯1回限り**) | サーバー側フラグで一度きり保証(再インストールしても再付与されない) |

- 「3日間の Pro 体験」は StoreKit(Apple の決済)を経由しない、アプリ内の期間限定権利付与(promo entitlement)。**体験終了後、自動的に有料プランへ課金・移行することはない**(規約 第6条の根拠。実装上、体験終了はローカルの期限判定のみで、Apple 側のサブスクリプション契約は一切発生しない)。
- 招待が成立する定義 = 「被招待者が紹介コードを入力し、上記の3回分特典が有効に付与された時点」。レビュー投稿・評価・SNS投稿は成立条件に一切含まれない(規約 第7条の根拠)。

## 2. 不正防止(規約 第8条の根拠)

- 自分自身のコードは入力不可(自己紹介の無効化)。
- 同一端末は同一コードを重複して兑換不可。
- 1端末は生涯1回のみ「招待された側」になれる(2つ目以降のコード入力は成立しない)。
- 同一 IP からの兑換成功件数に日次上限を設ける(不正な連続兌換の抑制)。
- 当社が不正取得(自己紹介、複数端末・再インストールによる反復取得等)と判断した場合、特典の無効化・付与停止を行う場合がある。
- 紹介コードの利用回数に、不正防止の観点から合理的な上限を設ける場合がある(具体的な回数は今後確定 — 2026-07-22 時点で未確定、`openTodos` 参照)。

## 3. 収集データ(規約 第9条の根拠)

| データ | 保存形式 | 目的 |
|---|---|---|
| 端末識別子(identifierForVendor 相当) | そのまま(照合用) | 特典の重複付与防止、不正防止 |
| IP アドレス | ハッシュ値のみ(平文は保存しない) | 同一 IP からの不正な連続兌換の検知 |

- 保存期間は不正防止の目的達成に必要な範囲に限定し、期限到来後は自動的に削除される(具体的な保存期間は運用上の設定であり、規約には期間の断定的な数値を記載しない。プライバシーポリシー側で必要に応じて補足)。
- 本項目は既存のプライバシーポリシー(`backend/public/legal/privacy/{lang}.html`)にも追記が必要(本タスクのスコープ外 — 別途対応)。

## 4. 規約に書かない/書けない事項(注意)

- **金額換算は規約に書かない**:特典は自社サービスの利用権であり、景品表示法上の総付景品規制(取引価額1,000円未満の場合、上限200円)との整合は社内検討事項として別途管理する。規約自体には円換算の記載をしない(`complianceOutline` の方針どおり)。
- **コード単位の総上限回数(MAX_REDEEMS_PER_CODE)は未確定**。確定後は本ドキュメントと規約第4条・第8条を同時に更新すること。
